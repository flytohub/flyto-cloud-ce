# MCP Studio and Client Setup

MCP Studio exposes saved workflows as Model Context Protocol tools and gives an
operator one place to test, connect, and audit them.

## Tool Contract

Flow discovers a workflow when its steps contain:

```yaml
- id: mcp_trigger
  module: flow.trigger
  params:
    trigger_type: mcp
    tool_name: collect_release_notes
    tool_description: Collect release notes for a repository
    config:
      input_fields:
        - name: repository
          type: string
          description: Repository owner and name
          required: true
```

The trigger input fields become the MCP tool's JSON Schema. If no fields are
declared, the tool accepts one generic object named `input`.

Changing a tool name or required field can break existing agent prompts and
client assumptions. Treat the trigger as a public API: add fields compatibly,
version breaking contracts, and test callers before removal.

## Tool Call Flow

```text
MCP tools/list
    -> discover MCP-triggered workflows
    -> return generated schemas and Flyto2 audit metadata

MCP tools/call
    -> resolve tool name to source workflow
    -> pass validated arguments into the local execution API
    -> wait for completion
    -> return result or structured failure
```

The MCP server keeps a short tool cache and refreshes it when needed. A saved
workflow remains the canonical source.

## Local Client Setup

Keep the Flow application running on <http://127.0.0.1:9000>.

### Codex

Add the configuration generated by **MCP Studio > Connect**. Its local form is:

```toml
[mcp_servers.flyto2-flow]
command = "python"
args = ["-m", "mcp_server"]
cwd = "/absolute/path/to/flyto-flow/src/ui/web/backend"

[mcp_servers.flyto2-flow.env]
FLYTO_BACKEND_URL = "http://127.0.0.1:9000"
```

Use an absolute `cwd` and a Python environment containing the backend
requirements.

### Claude Code or Another Stdio Client

```json
{
  "mcpServers": {
    "flyto2-flow": {
      "command": "python",
      "args": ["-m", "mcp_server"],
      "cwd": "/absolute/path/to/flyto-flow/src/ui/web/backend",
      "env": {
        "FLYTO_BACKEND_URL": "http://127.0.0.1:9000"
      }
    }
  }
}
```

The stdio bridge rejects a non-loopback `FLYTO_BACKEND_URL`.

### Streamable HTTP

The endpoint is:

```text
http://127.0.0.1:9000/api/mcp
```

POST clients must accept both `application/json` and `text/event-stream` as
required by the transport implementation. The endpoint supports JSON-RPC MCP
initialization, tool listing, and tool calls. A GET request returns setup
metadata; it is not an SSE stream.

## Access Model

### Loopback

Requests whose client and host are loopback may use the endpoint without an
account. This keeps local desktop and command-line clients simple.

### Non-Loopback

Non-loopback requests fail closed unless `FLYTO_FLOW_MCP_TOKEN` is configured.
Clients then send:

```text
Authorization: Bearer <operator-token>
```

Browser calls also pass an origin check. Add only reviewed origins to
`FLYTO_MCP_ALLOWED_ORIGINS`.

A token alone is not a complete internet-exposure design. Use TLS, an
authenticated reverse proxy, network policy, rate limits, secret rotation, and
log review.

## Audit Metadata

Each generated tool includes additive `_meta` fields:

| Field | Meaning |
| --- | --- |
| `flyto2/source` | Source type, workflow ID, and workflow name |
| `flyto2/contractVersion` | Version of the workflow-tool metadata contract |
| `flyto2/fingerprint` | SHA-256 of the source identity, tool name, and input schema |
| `flyto2/riskLevel` | Operator-defined risk, or the explicit default |
| `flyto2/approvalPolicy` | Workflow-defined approval policy, or the explicit default |
| `flyto2/evidenceRefs` | Evidence references declared by the trigger |

The fingerprint identifies contract drift; it is not a signature and does not
prove who authored the workflow.

## Design Guidance

- Give each tool one clear outcome and an action-oriented name.
- Write descriptions for agent selection, not for internal implementation.
- Declare typed inputs and mark only truly mandatory fields as required.
- Return a compact useful result; keep verbose diagnostics in evidence.
- Put destructive, financial, publishing, or access-changing steps behind an
  operator checkpoint.
- Never embed secrets in workflow JSON, examples, issue reports, or screenshots.
- Test failure and timeout paths, not only the successful call.

## Diagnostics

Inspect local MCP status:

```bash
curl --fail http://127.0.0.1:9000/api/mcp/status
```

If a tool is absent, verify the saved workflow contains the correct trigger and
tool name. If a call fails, reproduce it in the builder and inspect execution
history, evidence, and external-service connectivity.
